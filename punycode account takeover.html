The Most Underrated Account Takeover Using Punycode IDN Attacks

One invisible character in an email address. No phishing. No victim interaction. Just a character swap the server never caught.

ethicalrohit · 5 min read · punycode · idn · account-takeover

Introduction

Internationalized Domain Name (IDN) Homograph Attacks take advantage of characters from different languages that look nearly identical. For example, the Latin "a" and the accented "à" look almost the same to the human eye — but they are completely different characters under the hood.

When applications fail to handle this properly, an attacker can register using a lookalike email, trigger a password reset for the victim's original account, and log straight in. No clicks required from the victim.

The Core Idea
// What you see admin@gmail.com // victim's real email — Latin "a" àdmin@gmail.com // looks identical — but "à" is a different character
// Punycode encoded form xn--dmin-fsa@gmail.com // what the server actually receives
// The bug — server treats both as the same account admin@gmail.com == àdmin@gmail.com → Account Takeover

What Is Punycode?

Punycode is a way to represent special Unicode characters — like accented letters — using only normal ASCII text, so that domain name systems can process them.

For example, àdmin@gmail.com (with the accented "à") becomes xn--dmin-fsa@gmail.com in Punycode. You can verify any character using punycoder.com.

To most users, admin@gmail.com and àdmin@gmail.com look identical. But to the system, they are two completely different email addresses. The attack works because many applications don't normalize them consistently — signup treats them as different, but password reset treats them as the same. That mismatch is the bug.

How to Check if a Website Is Vulnerable

Before running the full attack, you first need to confirm the site is actually vulnerable. Here's the exact check — it takes under 2 minutes.

  • 1
    Register a normal account

    Sign up on the target site with a completely normal email address. For example:

    Step 1 — Normal signup
    admin@gmail.com

    Complete the signup. Log in. Confirm it works. Log out.

  • 2
    Try to register again using the Punycode version

    Go back to the signup page. Intercept the request in Burp Suite. Replace the email with the Punycode lookalike — swap "a" with "à":

    Step 2 — Punycode signup attempt (via Burp)
    àdmin@gmail.com // visually same, but different character

    Forward the request and look at the response carefully.

  • 3
    Read the response — this tells you everything
    What the server response means
    Response: "Email already exists" or "Account already registered" → VULNERABLE ✓ Server normalized your Punycode and matched the original account
    Response: Account created successfully → NOT vulnerable here. Server stored both as separate, different accounts.

    If you see "email already exists" — the site is vulnerable and you can proceed with the full account takeover below.

// why this works as a check

When the server says "email already exists" for your Punycode version, it means the backend normalized àdmin@gmail.com and matched it against the already-registered admin@gmail.com. That same normalization flaw is exactly what lets you hijack the password reset flow next.

Tools Required

  • Burp Suite — to intercept and modify HTTP requests
  • Burp Collaborator — acts as your fake SMTP server to receive password reset emails
  • punycoder.com — to generate the Punycode version of any character
// important note

Browsers like Chrome automatically encode special characters before they leave your machine. That's why you must use Burp Suite to intercept the request and replace the email field manually — otherwise the special character gets encoded and the attack won't work.

Step-by-Step Attack Walkthrough

  • 1
    Register with a normal email

    Go to the target website and sign up. Use Burp Collaborator as your email callback server so you can receive any emails the app sends you.

    Normal signup email
    admin@gmail.com.<your-collaborator-id>.oastify.com

    Log in to confirm the account works, then log out.

  • 2
    Sign up again using a Punycode lookalike email

    Go to signup again. Intercept the request in Burp Suite. Replace the email field with your Punycode version — swap "a" with "à":

    Punycode email — send via Burp, not the browser
    // "a" replaced with "à" (accented a) àdmin@gmail.com.<your-collaborator-id>.oastify.com

    If the server responds with "Email already exists" — vulnerability confirmed. The server normalized your Punycode email and matched it to the first account.

  • 3
    Trigger a password reset using the Punycode email

    Go to the Forgot Password page. Intercept the request in Burp. In the email field, submit the Punycode version:

    Password reset — Punycode email
    àdmin@gmail.com.<your-collaborator-id>.oastify.com

    The server normalizes the Punycode email, looks up the first account, and sends a password reset link to your Collaborator inbox.

  • 4
    Receive the reset link in Burp Collaborator

    Open Burp Collaborator and poll for incoming SMTP callbacks. The password reset email will arrive there. Copy the reset link from the email body.

  • 5
    Reset the password and log in as the victim

    Open the reset link in your browser and set a new password. Now log in using the original non-Punycode email with your new password:

    Account taken over
    // Login with the ORIGINAL email — no Punycode Email: admin@gmail.com.<collaborator-id>.oastify.com Password: <the new password you just set>
    // ✓ Logged in. Full account access. Zero victim interaction.

Why This Works

The root cause is simple. The application uses different normalization logic in different parts of the app. The signup flow stores the email one way. The password reset flow looks it up another way. That inconsistency is the entire vulnerability.

The signup flow and the password reset flow don't agree on whether two emails are the same. That disagreement is the bug.

No scanner will find this. There's no error message, no suspicious payload, no unusual response code. The login looks completely normal from the server's perspective. That's what makes it dangerous — and why programs pay well for it.

How to Fix It

Vulnerable
// Signup stores raw Unicode email as-is store(raw_email)
// Password reset normalizes at lookup time — mismatch! user = db.find( normalize(input) )
Correct Fix
// Normalize ONCE when the email first enters the system email = IDN.toASCII(input).toLowerCase().trim()
// Block Punycode prefixes if your app doesn't need international domains if "xn--" in email: reject("Invalid email format")
// Same normalized form used in signup, login, and reset — no mismatch store(email)

Conclusion

Punycode IDN attacks are one of the most underrated and cleanest account takeover techniques in bug bounty hunting. The victim does nothing. There is no phishing link to click, no social engineering, no suspicious activity. Everything happens silently through a normal-looking password reset flow.

Most hunters miss this because it doesn't come from running tools — it comes from understanding how an application handles the same piece of data across different flows. Once you see the pattern, you start finding it everywhere.

"Automated tools test what's written.
Manual testing exposes what's assumed."

// the bug is in the assumption. it always is.
