OpenID Connect Account Takeover Identity Binding Google SSO

I Found an Account Takeover by Reading a Spec, Not Running a Scanner

The login looked completely normal. The bug was one line of wrong logic. That's exactly what made it dangerous.

ethicalrohit · 6 min read · bug bounty · oidc · real bugs

Most bug bounty content is a recycling loop. Same recon tips. Same payload lists. Same "how I found XSS" stories that sound deep but don't actually teach you how real systems work.

I was stuck in that loop. Past beginner level, but not consistently finding anything meaningful. I had the workflow down — auth tests, edge cases, common web bugs — but it all felt surface-level.

The shift happened when I stopped reading writeups and started reading actual specs. This bug came directly from that.

Introduction

The target supported Sign in with Google through OpenID Connect. Standard stuff — user clicks the Google button, Google authenticates them, app receives identity data, user gets logged in.

At first glance, nothing looked strange. But I kept reading the OIDC specification, and one thing kept nagging at me: the application was treating the user's email address as their identity.

Email is not the stable identity in OpenID Connect. That sounds like a small detail. It's the entire bug.

Email is a mutable attribute — useful, displayable, storable. But it is not permanent. It can be reassigned, recycled, or returned under a completely different real-world owner. The spec is explicit about this. The application was not following it.

The Part Most Implementations Get Wrong

When an OpenID Connect provider authenticates a user, it returns claims inside an ID token. Most developers see the payload and immediately focus on one field:

ID Token Claims
email = "admin@clinic-example.com" // ← most devs look here first

And the logic that follows feels natural: same email = same user. It's wrong.

The stable identity in OIDC is the combination of two claims:

Stable Identity — OIDC Spec
iss // issuer — who is asserting this identity
sub // subject — unique, permanent ID for this user at that issuer

// This pair should bind the local account. Not email.

What the Vulnerable Logic Looked Like

// what the app was doing — wrong

Vulnerable
// Step 1 — Google returns identity data
email = "admin@org.com"

// Step 2 — App queries by email string
SELECT * FROM users WHERE email = 'admin@org.com'

// Step 3 — Logs that Google user into the found account
login(found_account) // ← account takeover lives here

// what it should have done — correct

Correct
// Step 1 — Google returns identity data
iss = "accounts.google.com"
sub = "1049302847362810" // unique, stable, never changes

// Step 2 — App queries by stable identity pair
SELECT * FROM users WHERE iss = ? AND sub = ?

// Step 3 — Only login if the stable identity matches
login(verified_account) // ← safe

The app wasn't asking "is this the same Google identity as before?" — it was asking "does this login return the same email string?" Those are not the same question.

Two Real-World Attack Scenarios

// scenario 01 — same org

Same email, completely different human

An org uses a shared mailbox — admin@clinic-example.com. User A logs in, creates an account. The app links it to the email string.

User A leaves. The org deprovisions their Google Workspace account and reassigns the same email to User B — a completely different person with a different sub value.

User B clicks "Sign in with Google." Google authenticates them correctly. The app sees the same email string and logs User B straight into User A's account. Full account takeover. No payload. No exploit. Just broken identity logic.

// scenario 02 — expired domain

Domain expires, mailboxes get recycled

An organization shuts down. Domain expires. Someone else registers the same domain and recreates old-looking mailboxes.

If the platform still has accounts tied to those email strings — and the login only trusts the email claim — the new mailbox owner just inherited access to someone else's account. Different person. Different country. Same email string.

Why This Bug Gets Underestimated

There's no script popup. No SQL error. No "hacked" splash screen. Nothing flashy to post anywhere.

// the dangerous part

The login looks completely legitimate from the application's point of view. Google authenticated the user correctly — it did its job. The break happens entirely in how the app chose to bind that authenticated identity to a local account. Nothing is broken at the provider level.

But the impact is real. If that account holds private messages, stored documents, privileged settings, or sensitive records — logging the wrong person in is a full confidentiality failure.

Why Tools Won't Find This

No scanner is going to flag this. No pattern to match, no response code to check. You can run every automated tool in your kit and walk away thinking the login flow is clean.

This class of bug only surfaces when you ask protocol questions, not payload questions:

→What exactly is this app using as the account key?
→Is it trusting email, or trusting iss + sub?
→What happens if the mailbox changes owners?
→What if the same email returns under a different Google identity?
→What if the domain expires and gets re-registered?

The Root Cause in One Sentence

The application used email as identity, when it should have used iss + sub. That single decision is enough to turn a standard Google login flow into an account takeover condition.

What the Correct Fix Looks Like

On first login via Google, store the iss and sub claims alongside the account. On every future login, match on those — not on email.

Email can still be stored. Displayed. Used for communication. But it should never be the thing that answers: is this the same person who created this account? That decision belongs to the stable provider identity.

The Bigger Lesson

Real vulnerabilities often don't come from clever payloads. They come from knowing how a protocol is supposed to work, then noticing the implementation quietly violates that contract. That gap is where strong bugs live:

  • OAuth misconfigurations
  • OIDC account confusion (this bug)
  • SAML validation gaps
  • JWT handling mistakes
  • Session binding flaws
  • Account recovery edge cases

None of these are as glamorous as XSS looks online. But they're often far more meaningful. And most of them become easier to spot once you stop only reading writeups and start reading the docs those writeups never mention.

"One teaches you what other people already found.
The other teaches you how to see what they missed."
